# Project Launch Playbook
## Пошаговый шаблон запуска automation / bot / scraper проектов

**Для кого:** все кто строит персональные инструменты автоматизации.
**Версия:** 2.2 — обновлено на основе реального опыта разработки HH Career Agent, май 2026.

---

## ГЛАВНЫЕ УРОКИ

> **Урок 1.** Документация до первого рабочего цикла — это фантастика.
> Код без проверки на реальных данных — это теория.
> Сначала заставь систему работать. Потом улучшай.

> **Урок 2. MVP — это не красивая архитектура.**
>
> Плохой MVP: 20 папок, 8 doc-файлов, 4 AI-агента, нет рабочего flow.
> Хороший MVP: 5 Python файлов, некрасивый код, но пользователь получает результат.
>
> MVP — это «система решает задачу хоть как-то».
> Красота приходит на итерации 2.

> **Урок 3. Скорость итерации важнее идеального дизайна.**
>
> На раннем этапе проекта цель MVP — не «написать правильно».
> Цель — как можно быстрее проверить: работает ли идея вообще?
>
> Быстрый feedback loop → быстрое обучение → быстрые решения.
> Медленный, "правильный" дизайн → поздний feedback → решения основанные на предположениях.
>
> Working loop важнее architecture purity.
> Feedback важнее abstractions.
> Скорость проверки гипотез важнее чистоты кода.

> **Урок 4. Первый реальный запуск разрушит твои гипотезы. Это запланировано.**
>
> Ты написал scoring → запустил → получил 15/15 нерелевантных результатов.
> Это не провал. Это и есть цель первого запуска.
>
> На этапе написания кода ты работаешь с предположениями о данных.
> Реальные данные покажут что ты не учёл: неправильные индустрии проходят,
> регион-фильтр не работает как ожидалось, ключевые слова дают не тот тип вакансий.
>
> **Calibration — это не исправление ошибок. Это обязательная часть разработки.**
>
> Ожидай что первый прогон потребует калибровки фильтров и весов.
> Не пытайся предусмотреть это заранее — невозможно.
> Запусти → посмотри → исправь → запусти снова.

## ФАЗА 0: Прояснение (30 минут)
### Сделай это ДО открытия редактора кода

**Ответь на 5 вопросов письменно (1–2 предложения каждый):**

```
1. Что делает система?
   Пример: "Парсит вакансии с hh.ru и присылает лучшие в Telegram"

2. Кто пользователь и какова его цель?
   Пример: "Я сама. Хочу не пропустить хорошие вакансии не проверяя сайт каждый день"

3. Что система получает на входе?
   Пример: "Ключевые слова для поиска, минимальная зарплата, мой chat_id"

4. Что система отдаёт на выходе?
   Пример: "Telegram-сообщение с названием, зарплатой, ссылкой и оценкой"

5. MVP: минимум что должно работать в первой версии?
   Пример: "Найти вакансию → отправить в Telegram → не слать дубли"
```

**Не делай на этом этапе:**
- Детальный roadmap на 5 этапов
- Архитектурные диаграммы
- Документацию для функций которых нет
- Планирование AI-агентов и будущих возможностей

## ФАЗА 1: Проверь внешние зависимости (30 минут)
### Прежде чем написать строку кода — убедись что API вообще работает

Каждый внешний сервис проверяй вручную перед тем как строить вокруг него код.

```bash
# Пример для HH API:
# Открой в браузере: https://api.hh.ru/vacancies?text=python&area=1
# → видишь JSON с вакансиями? Работает.
# → видишь {"type":"forbidden"}? Нужна регистрация приложения.

# Пример для Telegram:
# Создай бота у @BotFather → получи токен
# Проверь токен: https://api.telegram.org/botТОКЕН/getMe
# → видишь {"ok":true}? Работает.
# Напиши боту /start → проверь chat_id: https://api.telegram.org/botТОКЕН/getUpdates
```

**Правило: если хоть один API не работает — не пиши код вокруг него. Сначала разберись с API.**

## ПРАВИЛО: Сначала вручную, потом автоматизация

Это один из важнейших engineering принципов, который часто игнорируют новички.

**Если действие можно проверить вручную — сначала проверь вручную.**

| Что автоматизируешь | Как проверить вручную сначала |
|---------------------|-------------------------------|
| API запрос | Открыть endpoint в браузере |
| Telegram отправка | Послать сообщение через URL в браузере |
| OAuth flow | Пройти весь flow руками в браузере |
| Парсинг сайта | Открыть страницу, найти нужные данные глазами |
| Обработка файла | Открыть файл, проверить формат вручную |

**Почему это спасает часы работы:**

Когда что-то не работает в коде — есть три возможные причины:
```
1. Проблема в API / внешнем сервисе (токен, ключ, регистрация)
2. Проблема в Python коде (синтаксис, логика)
3. Проблема в архитектуре (неправильный подход)
```

Если сначала проверить вручную — ты исключаешь причину 1 полностью.
Тогда при ошибке в коде точно знаешь: проблема в твоём коде, а не во внешнем сервисе.

**Реальный пример из проекта:**
Мы писали scraper.py несколько часов, а HH API возвращал 403 потому что
нужна была регистрация приложения на dev.hh.ru.
Если бы сначала открыли endpoint в браузере — увидели бы это за 2 минуты.

> **Manual proof first. Automation second. Always.**

## WINDOWS-СПЕЦИФИКА: два обязательных gotcha
### Добавь в любой скрипт который запускается через Task Scheduler или терминал Windows

**Проблема 1 — кодировка (cp1251 vs UTF-8)**

Windows терминал и Task Scheduler используют кодировку cp1251 по умолчанию.
Эмодзи и кириллица в `print()` и `logging` упадут с `UnicodeEncodeError`.

```python
# Добавь в самом начале каждого .py файла который выводит текст:
import sys
sys.stdout.reconfigure(encoding="utf-8")
sys.stderr.reconfigure(encoding="utf-8")
```

**Проблема 2 — relative paths в Task Scheduler**

Когда Task Scheduler запускает скрипт — рабочая директория не та где лежит скрипт.
`open("data/seen.json")` → `FileNotFoundError`.

```python
# Добавь в if __name__ == "__main__": перед main():
import os
os.chdir(os.path.dirname(os.path.abspath(__file__)))
```

Это перед любым кодом который открывает файлы по относительному пути.

**Дополнительно в Task Scheduler:**
В поле **"Рабочая папка"** (Start in) — всегда указывай полный путь к папке проекта.
Без этого даже `os.chdir` не поможет если Python не найдёт сам скрипт.

## ФАЗА 2: Environment Setup (15 минут)
### Настрой окружение один раз и правильно — до любого кода

```bash
mkdir my-project && cd my-project
git init
git remote add origin https://github.com/username/repo.git
python -m venv venv
venv\Scripts\activate   # Windows
# source venv/bin/activate  # Mac/Linux
```

**Создай эти три файла сразу, до любого кода:**

**`.gitignore`** — первый файл который создаёшь:
```gitignore
__pycache__/
*.pyc
venv/
.env          # ← секреты НИКОГДА в git
logs/
data/*.json
data/*.log
*.pdf
*.odt
```

**`.env.example`** — шаблон без реальных значений (идёт в git):
```env
# Скопируй в .env и заполни реальными значениями
SERVICE_TOKEN=your_token_here
SERVICE_CHAT_ID=your_id_here
API_CLIENT_ID=your_client_id_here
```

**`.env`** — реальные значения (только локально, никогда в git):
```env
SERVICE_TOKEN=реальный_токен
SERVICE_CHAT_ID=реальный_id
API_CLIENT_ID=реальный_client_id
```

**Первый коммит — только структура:**
```bash
git add .gitignore .env.example
git commit -m "Initial project setup"
git push
```

## ПРАВИЛО: Health Check перед первым запуском
### Создай diagnose.py — проверку готовности системы

Перед тем как запускать основной бот — убедись что всё готово.
Создай отдельный скрипт `diagnose_runtime.py` который проверяет:

```python
"""Запуск: python diagnose_runtime.py"""
import os, sys, json
sys.stdout.reconfigure(encoding="utf-8")
os.chdir(os.path.dirname(os.path.abspath(__file__)))

from dotenv import load_dotenv
load_dotenv()

# Проверяй: токены, API connectivity, права на запись, целостность файлов памяти
checks = [
    ("TOKEN присутствует",       bool(os.getenv("SERVICE_TOKEN"))),
    ("logs/ доступна для записи", _check_write("logs/")),
    ("data/ доступна для записи", _check_write("data/")),
    ("API отвечает",              _check_api()),
]

for label, ok in checks:
    print(f"{'✅' if ok else '❌'} {label}")
```

**Почему это важно:**
Когда что-то не работает — есть три причины: токен, сеть, код.
`diagnose.py` исключает первые две за 10 секунд.
Без него тратишь час на отладку кода который не виноват.

**Правило:** запускай `diagnose_runtime.py` перед первым запуском и после любых изменений в `.env`.

## ПРАВИЛО: Сначала local, потом cloud

Не начинай с production infrastructure. Не масштабируй систему до первого работающего пользователя.

**Правильная последовательность:**

```
Сначала:                        Потом (когда реально нужно):
──────────────────────────      ──────────────────────────────
локальный запуск                Docker
локальные JSON файлы            база данных (PostgreSQL, SQLite)
локальные логи                  cloud logging
один пользователь               multi-user architecture
python main.py                  CI/CD pipeline
                                cloud deploy (Railway, Heroku, VPS)
```

**Почему это важно:**

Каждый слой инфраструктуры добавляет сложность и новые точки отказа.
На этапе MVP у тебя ещё нет понимания как именно система будет использоваться,
какая нагрузка будет реальной и что именно нужно масштабировать.

Поработай локально неделю — и эти вопросы ответят сами себя.

**Реальный пример:**
Docker и облачный сервер для личного Telegram-бота, который запускается раз в час — overengineering.
`python main.py` в терминале решает ту же задачу с нулевой инфраструктурой.

> **Local first. Cloud when local isn't enough.**

## ФАЗА 3: Минимальный рабочий pipeline (1–3 часа)
### Сделай так чтобы система делала своё главное действие

**Порядок создания файлов:**

```
1. config.py   — настройки + загрузка .env
2. source.py   — получить данные из источника
3. main.py     — вызвать source, распечатать в консоль

→ СТОП. Запусти. Данные пришли? Да → идём дальше. Нет → разбираемся здесь.

4. output.py   — отправить в Telegram / файл / email

→ СТОП. Запусти. Данные дошли? Да → идём дальше. Нет → разбираемся здесь.

5. memory.py   — дедупликация
6. main.py     — соединить всё в цикл
```

**Структура на этом этапе — и больше ничего:**
```
my-project/
├── config.py
├── source.py
├── output.py
├── memory.py
├── main.py
├── diagnose_runtime.py   ← добавь сразу
├── requirements.txt
├── .env
├── .env.example
└── .gitignore
```

Никаких `agents/`, `docs/`, `prompts/` пока. Создавай папку только тогда когда кладёшь в неё файл.

**Стандартные файлы в `logs/`:**
```
logs/agent.log           ← основной лог (RotatingFileHandler)
logs/run_history.jsonl   ← история запусков (одна строка JSON на запуск)
```

`run_history.jsonl` — append-only файл, одна строка на каждый запуск:
```json
{"timestamp": "2026-05-23T09:00:12", "duration_sec": 31.4, "sent": 2, "errors": 0}
```
Из него легко вытащить статистику. Из `agent.log` — нет.

## ПРАВИЛО: Сначала logs, потом abstractions

Перед тем как строить любую архитектуру — убедись что видишь что происходит.

```python
# Плохо — молчаливый код:
def fetch_vacancies(query):
    response = requests.get(url, params=params)
    return response.json().get("items", [])

# Хорошо — код который говорит что делает:
def fetch_vacancies(query):
    response = requests.get(url, params=params)
    print(f"[DEBUG] {query}: status={response.status_code}")
    if response.status_code != 200:
        print(f"[ERROR] Response: {response.text[:200]}")
        return []
    items = response.json().get("items", [])
    print(f"[DEBUG] Got {len(items)} items")
    return items
```

**Правила:**
- `print()` — нормально на этапе MVP, не стесняйся
- `logging.info()` — когда хочешь видеть flow
- `logging.error()` — когда нужно понять что сломалось

**Если не видишь что происходит — не строй архитектуру поверх темноты.**

Сначала добавь логирование → увидь что происходит → потом рефакторь.

## ПРАВИЛО: Debugging — это не отдельная стадия

Новички думают: **debugging = что-то пошло не так**.

На самом деле: **debugging = нормальная часть разработки**.

**Как выглядит настоящая разработка:**

```
написал код
    ↓
запустил
    ↓
увидел ошибку  ← это не провал, это информация
    ↓
понял систему лучше чем понимал до запуска
    ↓
исправил
    ↓
запустил снова
    ↓
(повторить)
```

Ошибки — это не отклонение от процесса. Ошибки и есть процесс.
Каждая ошибка говорит тебе что-то реальное о системе — то чего ты не знал до запуска.

**Что меняется в голове:**

| Установка новичка | Установка разработчика |
|-------------------|------------------------|
| «Ошибка = я облажался» | «Ошибка = система говорит мне правду» |
| «Нужно написать код без ошибок» | «Нужно быстро получить ошибку и понять её» |
| «Debugging — это потеря времени» | «Debugging — это основная работа» |
| «Запустил → должно работать» | «Запустил → смотрю что происходит» |

**Практически:**

Когда видишь ошибку — не паникуй. Прочитай сообщение целиком.
Оно почти всегда говорит что именно не так и где.

```
Traceback (most recent call last):
  File "scraper.py", line 34, in fetch_vacancies
    items = response.json().get("items", [])
requests.exceptions.JSONDecodeError: ...
```

Это не страшно. Это: «ответ от API пришёл не в формате JSON».
Следующий шаг: `print(response.text)` → посмотреть что реально вернул API.

> **Ошибка — это не конец. Это следующий вопрос.**

## Definition of Done — MVP считается готовым когда:

```
□ Данные реально приходят из источника (не mock, не тест-данные)
□ Output реально получает пользователь (сообщение пришло в Telegram)
□ Система переживает повторный запуск (запусти дважды)
□ Дубли не отправляются (второй запуск не шлёт те же данные)
□ При ошибке — понятное сообщение в лог, не падение всей системы
□ Новый человек может запустить проект за 15–30 минут по README
□ Проект можно поднять с нуля на другом компьютере
```

**Если хоть один пункт не выполнен — MVP не готов. Не переходи к следующему этапу.**

## ФАЗА 5: Добавление фич по одной (итерации по 1–2 часа)
### Каждая фича добавляется, тестируется, коммитится

**Одна фича за раз. После каждой — тест. Потом коммит.**

```
Итерация 1: Базовый фильтр (отсечь явный мусор)       → тест → коммит
Итерация 2: Scoring (оценить качество)                 → тест → коммит
Итерация 3: Форматирование вывода                      → тест → коммит
Итерация 4: Расписание (запуск каждые N минут)         → тест → коммит
Итерация 5: Объяснения (почему хорошая/плохая)         → тест → коммит
```

## ПРАВИЛО: Каждый новый слой должен решать реальную боль

Не добавляй слой "на будущее" или "чтобы было правильно".
Добавляй только тогда когда реально столкнулся с проблемой.

| Слой | Добавляй когда |
|------|---------------|
| Логирование | Непонятно что ломается |
| Memory / дедупликация | Начали приходить дубли |
| Фильтр | Слишком много нерелевантных результатов |
| Scoring | Нужно отличать хорошее от плохого |
| Форматирование | Сообщения трудно читать |
| Explain mode | Непонятно почему система приняла решение |
| AI агент | Обычные правила перестали справляться |
| База данных | JSON файл стал неудобен или медленен |

**Добавил слой "на будущее" — скорее всего выбросишь его при следующем рефакторе.**

## АНТИПАТТЕРНЫ

| Антипаттерн | Последствие | Как избежать |
|-------------|-------------|--------------|
| Документация до рабочего кода | Переписываешь после того как понял как работает | Пиши docs после MVP |
| Токены в коде | Утечка в git, cleanup, стыд | `.env` с самого начала |
| Сложная архитектура с первой итерации | Переусложнение, медленный старт | 5 файлов для MVP, потом растёт |
| Фичи до проверки базового pipeline | Отлаживаешь фичи поверх незаметных проблем | Тестируй каждый слой отдельно |
| Папки для будущих компонентов | Пустые директории, ложное ощущение прогресса | Создавай папку когда кладёшь в неё файл |
| README на 200 строк до рабочего кода | Устаревает быстрее чем дописываешь | 20-строчный README, расширяй по мере роста |
| Не проверил API вручную | Часы отладки кода вокруг несуществующего доступа | Manual proof first |
| Ранний refactoring | Переписываешь код который ещё не доказал что нужен | Сначала 3–5 запусков на реальных данных |
| Слой "на будущее" | Мёртвый код, лишняя сложность | Добавляй только когда есть реальная боль |
| Молчаливый код (нет логов) | Не понимаешь что происходит, строишь архитектуру вслепую | `print()` и `logging` с первой строки |
| Начать с cloud/Docker | Инфраструктурная сложность до рабочего кода | Local first, cloud when needed |
| «Ошибка = провал» | Избегаешь запуска, теряешь feedback | Запускай часто, ошибки — это информация |
| `while True` для личного scheduled бота | Привязан к терминалу, падает незаметно, нет истории | Single-run + Task Scheduler / cron |
| Нет diagnose скрипта | Час отладки кода который не виноват (проблема в токене) | `diagnose_runtime.py` запускай первым |
| Нет `sys.stdout.reconfigure` на Windows | `UnicodeEncodeError` при первом запуске с эмодзи | Добавляй в начало каждого скрипта |
| Нет `os.chdir` для scheduler | Relative paths ломаются в Task Scheduler | `os.chdir` в `if __name__ == "__main__":` |

### main.py — single-run (рекомендованный паттерн)

Для личных ботов: **не используй `while True`**. Запускай один цикл → выход.
Расписание делегируй Task Scheduler (Windows) или cron (Mac/Linux).

```python
"""AgentName — что делает. Запуск: python main.py"""
import os
import sys
import time
import json
import logging
import traceback

sys.stdout.reconfigure(encoding="utf-8")
sys.stderr.reconfigure(encoding="utf-8")

from config import ALERT_MENTION   # "@username" или ""
from source import fetch_data
from processor import filter_and_score
from memory import load_seen, save_seen, filter_new
from output import send_result, send_text


def run_cycle(seen_ids: set) -> tuple[set, int]:
    items   = fetch_data()
    new     = filter_new(items, seen_ids)
    results = filter_and_score(new)
    sent = 0
    for item, score in results:
        send_result(item, score)
        sent += 1
    return seen_ids | {item["id"] for item in new}, sent


def _write_run_history(ts, duration, sent, error=None):
    record = {"timestamp": ts, "duration_sec": round(duration, 1),
              "sent": sent, "errors": 1 if error else 0, "error_msg": error}
    os.makedirs("logs", exist_ok=True)
    with open("logs/run_history.jsonl", "a", encoding="utf-8") as f:
        f.write(json.dumps(record, ensure_ascii=False) + "\n")


def main():
    logging.basicConfig(level=logging.INFO,
                        format="%(asctime)s [%(levelname)s] %(message)s")
    log = logging.getLogger(__name__)

    start = time.time()
    log.info("Agent started")

    try:
        seen = load_seen()
        seen, sent = run_cycle(seen)
        save_seen(seen)
    except Exception as e:
        elapsed = time.time() - start
        tb = traceback.format_exc(limit=4)[-500:]
        log.error(f"Critical error: {e}", exc_info=True)
        send_text(f"{ALERT_MENTION} 🚨 ERROR\n{type(e).__name__}: {e}\n\n{tb}")
        _write_run_history("now", elapsed, 0, str(e))
        raise SystemExit(1)

    elapsed = time.time() - start
    log.info(f"Done in {elapsed:.1f}s | sent: {sent}")
    _write_run_history("now", elapsed, sent)

    if sent > 0:
        send_text(f"✅ Готово — отправлено: {sent} | {elapsed:.0f}с")


if __name__ == "__main__":
    os.chdir(os.path.dirname(os.path.abspath(__file__)))  # scheduler-safe
    main()
```

**Почему single-run лучше `while True` для личных инструментов:**

| | while True | Single-run + Scheduler |
|---|---|---|
| Требует открытый терминал | Да | Нет |
| Падает незаметно | Да (молчит) | Нет (scheduler перезапустит) |
| Видно в логах каждый запуск | Сложно | Да (run_history.jsonl) |
| Ресурсы когда не работает | Держит процесс | Ноль |

### Telegram — не только output, но и error alert

Telegram — идеальный канал для уведомлений об ошибках.
Добавь в `config.py`:
```python
ALERT_MENTION = "@your_username"  # "" чтобы отключить
```

Когда бот падает ночью — ты получишь push-уведомление с @упоминанием,
которое пробивает мьют. Без этого узнаешь об ошибке только когда заметишь
что ничего не пришло.
