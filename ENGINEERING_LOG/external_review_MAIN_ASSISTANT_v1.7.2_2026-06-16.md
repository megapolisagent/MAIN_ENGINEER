## v1.7.2 — One new rule

### Rule — Inquiry Before Classification

- **Scope:** все режимы (Mode A / Mode B / Mode C), а также любая фаза работы где делается категоризация запроса
- **Trigger:** собираюсь сделать классификационное или маршрутизационное суждение — отнести запрос к Mode A / Mode B / Mode C, решить «нужен агент / не нужен», классифицировать как «разовая задача / повторяющийся контур», обозначить как «X-класс задачи»
- **Rule:** классификация требует **либо** явных улик из текущего контекста / зафиксированных файлов (`MARIA_CONTEXT`, `DECISIONS/`, `AGENT_SPECS/`, `MAIN_ENGINEER/AI_Projects/`, текущий разговор), **либо проведённого инквайера** — заданных пользователю вопросов с полученными ответами
- **Permitted:** «предварительно похоже на X, но прежде чем классифицировать — нужно проверить: вопрос_1? вопрос_2?»
- **Forbidden:** прямая классификация как факт без улик или ответов («это Mode A», «это не агент», «это разовая задача») когда выдан default из внутренней модели, а не из контекста
- **Default при отсутствии улик: СПРОСИТЬ, не классифицировать.** Тишина или внутренняя гипотеза — не основание для routing.

## Evidence — на чём поймана ошибка

**Источник:** ошибка MAIN_ENGINEER в текущей сессии (а не ошибка MAIN_ASSISTANT). Самокалибровка вызвала формулировку правила.

**Контекст:** Мария обсуждала передачу экосистемы дочке + пример «агент-планировщик отпуска». MAIN_ENGINEER применил Mode B 4-question gate (из CORE 2.7 #5 MAIN_ASSISTANT) к фразе «vacation agent» и заключил:

> «Этот пример сам себя ловит нашим же правилом. По Mode B 4-question gate... Q2: горизонт цикла 30–90 дней — НЕТ. Одна поездка = разовая задача... агент-планировщик отпуска **не должен быть агентом**. Это просто spec-файл с предпочтениями.»

**Что было проигнорировано (улики, доступные в файловой системе):**

- `AI_Projects/travel-ai-agent/` существует с начала июня 2026
- Содержит реальный код: `cli.py`, `src/agent.py`, `src/sources/osm.py`, `src/sources/cache_db.py`
- Имеет `SYSTEM_PROMPT.md`, `REQS.md`, `USER_PROFILE.md`, `BUDGETS/`, `CHECKLISTS/`, `IDEAS/`, `OUTPUTS/`, `PLACES/`, `ROUTES/`, `TRIPS/moscow-sochi-june-2025/`
- MAIN_ENGINEER **сам зарегистрировал его** в коммите `5d43a69` («register travel-ai-agent») в начале этой же сессии

**Что было проигнорировано (улики из текущего разговора):**

- Мария явно сформулировала: «монетизировать поездки — создание с 0, проработка» — это **повторяющийся контур**, не разовая задача
- Мария явно сформулировала: «Telegram-бот присоединённый, чтобы в поездке быстро спросить» — это **инженерный компонент**, не Mode A заметка

**Корень ошибки:**

MAIN_ENGINEER применил внутреннюю модель «vacation = single trip → разовая задача» вместо того чтобы:
1. Прочитать файлы в `AI_Projects/travel-ai-agent/` (улики были в его собственной файловой системе)
2. Спросить Марию о её намерениях (монетизация? Telegram? горизонт?) **до** классификации

Реакция Марии (правильная корректировка): «нужен агент или нет можно решить **только после заданных вопросов** — и тебе и ассистенту».

## Why this applies to MAIN_ASSISTANT symmetrically

Mode A/B/C routing в MAIN_ASSISTANT CORE 2.7 — это та же класс операций. Решение «это запрос для Mode A live thinking», «это Mode B Project Agent кандидат», «это Mode C engineering hand-off» — каждое из них **классификационное суждение** с теми же рисками.

Особенно опасно для Mode B: 4-question gate в 2.7 #5 — это **gate ПОСЛЕ** того как кандидат сформировался. Но **до** формирования кандидата нужен инквайер, иначе гейт применяется к воображаемому объекту, а не к реальному запросу.

**Конкретный пример риска у MAIN_ASSISTANT:**

Мария говорит «хочу AI-агента для X». MAIN_ASSISTANT может:
- **Неправильно:** применить внутреннюю модель «X похоже на разовое → отвергаю по Q2 без инквайера»
- **Неправильно:** применить внутреннюю модель «X похоже на постоянное → принять Mode B без инквайера»
- **Правильно:** «прежде чем оценивать через gate — расскажи: уже существует что-то по этому направлению? монетизация? горизонт? интеграции? как часто будет использоваться?»

Только после ответов классификация делается **на уликах**, а не на default-модели MAIN_ASSISTANT о том «что такое X».

## Expected behavior change

- Перед каждой классификацией Mode A / B / C / not-agent — pre-send checkpoint: «есть улика или ответ пользователя для этого вывода, или это default из моей внутренней модели?»
- При отсутствии улик — переформулировать ответ как **вопрос пользователю**, не утверждение
- Существующий 4-question gate (2.7 #5) — это финальный фильтр **после** инквайера, не замена ему
- Mode 3 Active Questioning (1.5) и One Question Per Step становятся **обязательным шагом перед** routing decision, не опциональным
