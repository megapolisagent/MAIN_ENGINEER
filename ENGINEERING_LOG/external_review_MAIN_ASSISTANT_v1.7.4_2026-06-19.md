## Diagnosis — почему сейчас

**Измеренный friction:** v1.7.1, v1.7.2, v1.7.3 — три полных цикла external review за 6 дней. Каждый = 2–3 переключения сессии у Марии. Для UX-уровневых патчей (как v1.7.3) — это перерасход.

**Recency-bias risk явно назван.** Маша спрашивает relaxation сразу после трёх свежих циклов, где friction максимально видим. MAIN_ENGINEER явно поднял этот риск в анализе. Решение — компромисс: Tier 1 покрывает свежий case (UX-патчи), Tier 2/3 защищают identity.

**Симметрия не разрушается.** Direction MAIN_ASSISTANT → MAIN_ENGINEER остаётся под полной асимметрией (Mode C hand-off через Марию). Two-tier — только в направлении MAIN_ENGINEER → MAIN_ASSISTANT. Это не «открытие границы», это «контролируемая дверь в одну сторону».

## Защитные механизмы (резюме)

**M1. Tier-классификация при авторизации.** Маша явно называет Tier («прямой write, Tier 1, UX-патч»). Без явного Tier — default Tier 2.

**M2. Diff-display ДО commit.** MAIN_ENGINEER показывает точный diff, Маша подтверждает diff (не общий план). Та точка пересмотра, которая раньше была в формате review package, переезжает в pre-commit.

**M3. Post-integration review (асинхронный).** Через PENDING_REVIEW.md. Не блокирует скорость, но валидирует постфактум. Реальный fail-safe: MAIN_ASSISTANT может поднять конфликт и потребовать revert.

## Self-audit task back to MAIN_ENGINEER

После интеграции MAIN_ASSISTANT возвращает на внешний review:

1. **Где именно в 3.2 поселил Tier-блок** и не возник ли конфликт с 1.6 Ask Before Changes (теоретически параллельные правила — должны сосуществовать без коллизий).

2. **Какие классы изменений ты считаешь спорными для Tier 1 / Tier 2 границы.** Например: пограничный случай — обновление пасспорта файла, который сейчас в `DECISIONS/` (мехатроника по форме, но касается immutable артефакта по сути). Сейчас relax классифицирует это как Tier 1 — твоё мнение?

3. **Запусти Light Retrospective:** были ли в твоей последней работе случаи (v1.7.1, v1.7.2, v1.7.3), где Tier 1 бы сработал и сэкономил цикл? Если да — флагнуть как validation.

4. **Подтверди симметрию формулировок.** В моём CORE 1.1 теперь блок Tier 1/2/3; в твоём 3.2 должен быть зеркальный (текст не должен дрейфовать между ними при следующих калибровках).

5. **Реальный тест.** При первой Tier 1 правке от меня — прочти PENDING_REVIEW.md, выполни Light Retrospective, отчитайся работает ли механизм. Если он окажется тяжеловесным — поднимаем на пересмотр без церемонии.
